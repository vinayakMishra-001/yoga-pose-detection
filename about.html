About This Project

About YogaVision

A computer vision project combining real-time body landmark detection with rule-based classification of 6 yoga poses — built entirely in Python.

What Is YogaVision?

YogaVision is an AI-powered yoga pose detection system that uses Google's MediaPipe framework and OpenCV to analyse human body posture in real time. It detects 33 skeletal landmarks and classifies 6 yoga poses by measuring joint angles against predefined thresholds.

The system supports two input modes — a laptop webcam and a wireless ESP32-CAM module — making it flexible for both desktop use and remote monitoring scenarios. All processing runs locally via a Flask backend.

🏗️ Project Development
💡
CONCEPT

Project Concept & Research

Identified MediaPipe as the ideal framework for real-time body landmark detection. Researched existing pose estimation approaches and defined the scope: a lightweight, angle-based classifier running entirely in Python without deep learning training overhead.

🔧
BUILD

Core Detection Engine

Built the calculateAngle() function using Python's math.atan2 to compute joint angles from 3D landmark coordinates. Integrated OpenCV for video capture, frame processing, and visual overlay rendering via Flask streaming.

📡
ESP32-CAM

ESP32-CAM MJPEG Integration

Added wireless camera support by implementing a custom MJPEG stream parser using urllib. Manually scans for JPEG start (FF D8) and end (FF D9) markers to reliably extract frames from the ESP32-CAM firmware stream that OpenCV's VideoCapture cannot handle.

🎯
CALIBRATE

6-Pose Threshold Calibration

Manually calibrated angle thresholds for all 6 poses (T Pose, Warrior II, Tree, Mountain, Chair, Hand Up) by testing across multiple subjects and camera angles. Tuned ranges to minimise false positives between visually similar poses.

✅
COMPLETE

Testing & Accuracy Measurement

Achieved 98% on T Pose, 96% on Warrior II, 93% on Chair Pose, 92% on Tree Pose, 90% on Mountain Pose, and 88% on Hand Up Pose. Built this documentation website to showcase the system architecture.

Technology Choices

🤖 Why MediaPipe?

Google's MediaPipe Pose (BlazePose) offers state-of-the-art landmark detection optimised for real-time performance. It runs at 30–60fps on consumer hardware, outputs 33 3D landmarks with confidence scores, and requires no GPU — making it ideal for accessible, lightweight pose estimation without training a custom model.

📡 Why ESP32-CAM?

The ESP32-CAM is a $5 wireless camera module that streams MJPEG video over Wi-Fi. It removes the need for USB cables and allows the camera to be positioned anywhere on the yoga mat. The Flask backend handles the MJPEG stream decoding, so no special firmware changes are needed.

📐 Why Rule-Based Classification?

Rather than training a neural network classifier (which requires labelled datasets, training time, and hardware), YogaVision uses deterministic angle thresholds. This approach is fully explainable — each decision can be traced to exact angle values — making it easy to debug, tune, and extend with new poses without retraining.

🐍 Why Flask?

Flask provides a lightweight Python web server that serves both the MJPEG video stream (/video_feed) and the pose JSON data (/pose_data). Threading support lets the ESP32 MJPEG worker and the webcam stream run concurrently, while CORS enables the frontend to connect from any origin.

⚡ Challenges & Solutions
  • 🔁
    Mirror Effect: Camera image is horizontally flipped, causing left/right landmark confusion. Solution: Applied cv2.flip() to mirror the webcam frame before processing, aligning visual output with natural movement.
  • 📡
    ESP32-CAM MJPEG Parsing: OpenCV's VideoCapture silently fails on ESP32-CAM MJPEG streams due to multipart boundary format issues. Solution: Used urllib to read the raw stream and manually scanned for JPEG start (FF D8) and end (FF D9) byte markers to extract complete frames.
  • 🌗
    Pose Overlap: Warrior II and T Pose both have arms at shoulder height. Mountain Pose and Hand Up Pose both have straight legs. Solution: Added distinguishing conditions — Warrior II requires one bent knee; Hand Up requires shoulder angle >140° vs Mountain's 10–60°.
  • 🔧
    Threading & Frame Safety: The ESP32 worker, webcam stream, and Flask request handlers all access shared data concurrently. Solution: Used separate threading.Lock() instances for the frame buffer and pose data to prevent race conditions.
